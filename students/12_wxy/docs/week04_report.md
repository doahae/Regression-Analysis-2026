# 线性回归求解器对比报告

## 实验环境

- **CPU**: Intel/AMD 多核处理器
- **内存**: 16GB+
- **操作系统**: WSL (Ubuntu)
- **Python版本**: 3.12

## 实验参数

| 实验场景 | 样本数 (N) | 特征数 (P) |
|---------|-----------|-----------|
| 低维场景 | 10,000 | 10 |
| 高维场景 | 10,000 | 2,000 |

## 求解器列表

| 求解器 | 类型 | 实现方式 |
|--------|------|---------|
| Custom Analytical | 解析解 | 正规方程 + `np.linalg.solve` |
| Custom GD | 数值解 | 批量梯度下降 (自实现) |
| Statsmodels OLS | 解析解 | 统计学派实现 |
| Sklearn Linear | 解析解 | 基于 `np.linalg.lstsq` |
| Sklearn SGD | 数值解 | 随机梯度下降 |

## 结果分析

### 1. 低维场景 (P=10)

- 所有求解器表现良好，MSE 约 0.0096
- 运行时间差异不大，都在 0.01 秒左右
- Custom GD 和 Analytical 精度几乎一致

### 2. 高维场景 (P=2000)

- **Custom Analytical**：仍能快速求解 (0.013s)，因为使用了 `np.linalg.solve` 而非显式求逆
- **Custom GD**：耗时 33.4 秒，显著慢于其他方法，因为每次迭代需要处理全部 10000 个样本
- **Statsmodels OLS** 和 **Sklearn Linear**：表现良好 (0.013s)
- **Sklearn SGD**：训练快 (0.0087s)，但预测结果数值爆炸，原因是高维数据未标准化导致梯度爆炸

## 思考题解答

### Q1: 在高维场景下，哪个 API 崩溃了或极其缓慢？

**Custom GD (批量梯度下降)** 耗时 33.4 秒，是其他方法的 **2500+ 倍**。

**原因**：
- 每轮迭代计算梯度需要 O(N×P) = 2000万次操作
- 迭代 2000 轮，总操作量约 400亿次
- 相比之下，正规方程只需要一次 O(P³) = 80亿次操作，但实际常数小得多

### Q2: 为什么 SGDRegressor 能在极短时间内完成任务？

**理论上** SGD 应该很快，因为：
- 每次只用 1 个样本计算梯度，单次 O(P)
- 每 epoch 更新 N 次，总操作量 O(epochs × N × P)

**但本实验** 中 SGD 出现数值爆炸，原因是：
- 高维数据 (P=2000) 特征尺度差异大
- 未使用 StandardScaler 标准化
- 学习率过大导致梯度爆炸

**解决方法**：在 SGD 前添加 `StandardScaler` 对特征进行标准化。

## 结论

1. **解析解方法**（Analytical、Statsmodels、Sklearn Linear）在高维下仍然快速稳定
2. **自实现的批量梯度下降** 在高维下非常慢，不适合实际应用
3. **SGD 需要配合特征标准化**，否则容易数值不稳定
4. 工业级实现（Sklearn）在稳定性和速度上都优于手写实现
